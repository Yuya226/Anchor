# Anchor SPEC

## 1. 目的

Anchor は、自分専用の iOS アプリである。

目的は、迷いやすい自分が、今やることに戻れる「アンカー」を持つこと。

このアプリは、完璧な人生設計をするためのアプリではない。選択肢が多すぎて動けなくなる状態を防ぎ、迷い・実行・無行動・棚上げ・代替行動を記録し、自分の内省を外部化・体系化するためのアプリである。

Anchor は、長文を書く日記アプリではない。予定していたこと、実際にやったこと、迷ったこと、棚上げしたこと、何もしなかったことを、選択・短文・時間ログによって軽く記録する「現代版の日記」である。

---

## 2. 設計思想

### 2.1 迷いを否定しない

迷いは失敗ではない。迷いは、選択肢の多さと向き合った記録である。

ただし、迷い続けて行動が止まる場合は、それを見える化し、扱える形にする。

### 2.2 迷いは解決せず、まず預ける

実行中に迷ったら、その場で結論を出そうとしない。

迷いログに保存し、今やることに戻る。

### 2.3 選ばなかった選択肢は捨てずに棚上げする

「切る」は、可能性を否定することではない。

今この瞬間に迷い続けるコストを減らすため、一時的に選択肢を閉じることである。

### 2.4 価値は、やりきった後に見える

やる前に意味を完全に証明しなくてよい。

一定時間取り組んだ後に、直接成果と副産物の両方を回収する。

### 2.5 実行前は理想に寄せ、実行後は現実を受け止める

実行前の画面では、選択肢を増やさない。

ホーム画面に表示するタスクは、今本当にすべきことだけに絞る。娯楽、逃避、代替行動、棚上げ中の選択肢を、実行前の主要選択肢として並べすぎない。

実行後、または中断後には、軽いフィードバック画面を表示する。そこで、本来のタスクができたのか、別のことをしたのか、何もしなかったのかを記録できるようにする。

この「予定と実際の差分」を記録することが、Anchor の重要な価値である。

### 2.6 最初は自分専用

競合調査、市場調査、ユーザー拡大、収益化は考えない。

まずは自分が毎日使える最小版を作る。

## 5. 画面構成

### 5.1 ホーム画面 / 今日のアンカー

毎日最初に見る画面。迷ったときに戻る場所。

表示するもの：

- 今日のアンカー
- 今の重点テーマ
- 次の一手
- 今日の積み上げ時間
- 今日の迷い回数
- 棚上げ中の項目の一部
- 作業開始ボタン
- 迷ったボタン
- チェックインボタン

役割：

- 今やることを思い出す
- 選択肢を増やさず、本当にすべきことだけを表示する
- 迷ったら記録して戻れるようにする

### 5.2 迷いログ画面

迷った内容を保存・確認する画面。

最初に保存するもの：

- content: String
- createdAt: Date

将来的に保存する可能性：

- relatedTheme: String
- intensity: Int
- result: String

最初の一覧画面は閲覧専用でよい。削除、編集、検索、分類、強さの記録は後回しにし、まずは「預けた迷いを見返せる」ことを優先する。

### 5.3 棚上げリスト画面

今はやらないが、捨てない選択肢を保存する画面。

保存するもの：

- 棚上げしたこと
- 理由
- 作成日時
- 見直し日

### 5.4 ActivitySession / 作業・行動タイマー画面

何にどれだけ時間を投下したかを記録する画面。

Anchor が記録するのは、勉強や作業だけではない。実際に起きた行動全体である。

最初はホーム画面内の簡易タイマーでもよい。

### 5.5 セッション終了後フィードバック画面

作業開始後、終了または中断したときに、軽いフィードバック画面を表示する。

確認する内容：

- 本来のタスクはできたか
- どのくらいできたか
- 別のことをしたか
- 何もしなかったか
- 迷いが影響したか
- 実際には何をしたか
- 一言メモ

最初の選択肢案：

- できた
- 少しできた
- 別のことをした
- 何もしなかった

「別のことをした」を選んだ場合、代替行動を選べるようにする。

代替行動の例：

- YouTube
- Netflix
- アニメ
- 映画
- 筋トレ
- 散歩
- 睡眠
- ポルノ
- 別の勉強
- バイト
- 何もしない
- その他

これはユーザーを責めるためではない。実際に時間がどこへ流れたのかを把握するためである。

### 5.6 チェックイン画面

1日の終わりに軽く記録する画面。

保存するもの：

- 達成度 1〜5
- 納得度 1〜5
- 迷いの強さ 1〜5
- 一言メモ
- 作成日時

## 7. 将来拡張：タスクとコンテキスト

タスクは、常に同じ条件で実行できるわけではない。実行可能性は、今の状況によって変わる。

例：

- 家にいる
- 移動中
- バイトの休憩中
- 授業中の空き時間
- スマホしか使えない
- PCが使える
- 5分だけある
- 30分以上ある
- 集中できる
- 疲れている

将来的には、タスクに実行コンテキストを持たせる。

例：

- 必要デバイス：スマホ / PC / どちらでも可
- 実行場所：家 / 外 / 移動中 / 授業中 / 休憩中
- 必要時間：5分 / 15分 / 30分 / 60分
- 必要集中度：低 / 中 / 高

これにより、今の状況で実行可能なタスクだけを出せるようにする。

ただし、MVPでは本格実装しない。まずは将来拡張しやすい設計に留める。

## 8. デザイン方針

- 落ち着いた雰囲気
- 自分を責めない言葉遣い
- 余白を多めにする
- カードUI中心
- 情報を詰め込みすぎない
- 1画面で「戻ってこられる感覚」を出す
- 派手な色や過度なアニメーションは不要

---

## 9. 文言方針

### 避ける表現

- 達成率が低いです
- もっと頑張りましょう
- 未完了タスクがあります
- 失敗しました

### 使いたい表現

- 迷いはここに預ける
- 次の一手に戻る
- 今日は迷いが多い日でした
- 記録すれば、次に使えます
- 今は考えるより進める時間です
- 棚上げして、今やることに戻る

## 11. Codexへの基本方針

- SPEC.md を読んだうえで実装する
- 変更範囲は小さくする
- 一度に多くの機能を実装しない
- 既存のUIを大きく崩さない
- まずビルドが通ることを優先する
- 外部ライブラリは使わない
- SwiftDataで保存する
- 複雑な抽象化はまだしない
- まず自分が使える最小実装を優先する

---

## 12. 現在の最優先体験

最優先は、以下の体験を作ること。

迷ったときにアプリを開く  
→ 今日のアンカーを見る  
→ 迷いをログに預ける  
→ 今やることに戻る  

この体験が成立するまでは、他の機能を増やしすぎない。
