# Anchor SPEC

## 目的

Anchor は、自分専用の iOS アプリである。

目的は、迷いやすい自分が、今やることに戻れる「アンカー」を持つこと。

このアプリは、完璧な人生設計をするためのアプリではない。
選択肢が多すぎて動けなくなる状態を防ぎ、迷い・実行・無行動・棚上げを記録し、自分の内省を外部化・体系化するためのアプリである。

## 設計思想

### 1. 迷いを否定しない

迷いは失敗ではない。
迷いは、選択肢の多さと向き合った記録である。

ただし、迷い続けて行動が止まる場合は、それを見える化し、扱える形にする。

### 2. 迷いは解決せず、まず預ける

実行中に迷ったら、その場で結論を出そうとしない。
迷いログに保存し、今やることに戻る。

### 3. 選ばなかった選択肢は捨てずに棚上げする

「切る」は、可能性を否定することではない。
今この瞬間に迷い続けるコストを減らすため、一時的に選択肢を閉じることである。

### 4. 価値は、やりきった後に見える

やる前に意味を完全に証明しなくてよい。
一定時間取り組んだ後に、直接成果と副産物の両方を回収する。

### 5. 最初は自分専用

競合調査、市場調査、ユーザー拡大、収益化は考えない。
まずは自分が毎日使える最小版を作る。

## 追加設計思想：現代版の日記としてのAnchor

Anchor は、長文を書く日記アプリではない。

Anchor は、

- 本来やる予定だったこと
- 実際にやったこと
- 迷ったこと
- 棚上げしたこと
- 何もしなかったこと
- 代わりに流れた行動

を軽く記録する、現代版の日記である。

文章で深く書くのではなく、選択・短文・時間ログによって、自分の行動と内省を外部化する。

## 実行前と実行後の役割分離

### 実行前

実行前の画面では、選択肢を増やさない。

ホーム画面に表示するタスクは、今本当にすべきことだけに絞る。

娯楽、逃避、代替行動、棚上げ中の選択肢を、実行前の主要選択肢として並べすぎない。

目的は、判断コストを下げ、今やるべきことに戻ることである。

### 実行後

実行後、または中断後には、軽いフィードバック画面を表示する。

そこで、本来のタスクができたのか、別のことをしたのか、何もしなかったのかを記録できるようにする。

実行前は理想に寄せる。
実行後は現実を受け止める。

この差分を記録することが、Anchor の重要な価値である。

## タスクとコンテキスト

タスクは、常に同じ条件で実行できるわけではない。

実行可能性は、今の状況によって変わる。

例：

- 家にいる
- 移動中
- バイトの休憩中
- 授業中の空き時間
- スマホしか使えない
- PCが使える
- 5分だけある
- 30分以上ある
- 集中できる
- 疲れている

将来的には、タスクに実行コンテキストを持たせる。

例：

- 必要デバイス：スマホ / PC / どちらでも可
- 実行場所：家 / 外 / 移動中 / 授業中 / 休憩中
- 必要時間：5分 / 15分 / 30分 / 60分
- 必要集中度：低 / 中 / 高

これにより、今の状況で実行可能なタスクだけを出せるようにする。

ただし、MVPではコンテキスト判定は本格実装しない。
まずは将来拡張しやすい設計に留める。

## 代替行動ログ

本来やる予定だったタスクの代わりに、実際に何をしたかを記録できるようにする。

これはユーザーを責めるためではない。
実際に時間がどこへ流れたのかを把握するためである。

記録対象の例：

- YouTube
- Netflix
- アニメ
- 映画
- 筋トレ
- 散歩
- 睡眠
- ポルノ
- 別の勉強
- バイト
- 何もしない
- その他

重要なのは、良い行動だけを記録しないこと。
実際に起きた行動を、そのまま記録する。

## セッション終了後フィードバック

作業開始後、終了または中断したときに、軽いフィードバック画面を表示する。

確認する内容：

- 本来のタスクはできたか
- どのくらいできたか
- 別のことをしたか
- 何もしなかったか
- 迷いが影響したか
- 実際には何をしたか
- 一言メモ

最初の選択肢案：

- できた
- 少しできた
- 別のことをした
- 何もしなかった

「別のことをした」を選んだ場合、代替行動を選べるようにする。

## ActivitySession 方針

作業時間ログは、単なる WorkSession ではなく ActivitySession として扱う。

理由：

Anchor が記録するのは、勉強や作業だけではない。
実際に起きた行動全体である。

ActivitySession は、以下を記録する。

- 本来やる予定だったタスク
- 開始時刻
- 終了時刻
- 所要時間
- 結果
- 実際にやったこと
- 代替行動カテゴリ
- メモ

MVPでは、まず以下だけでよい。

- title: String
- startedAt: Date
- endedAt: Date?
- durationSeconds: Int
- result: String?
- actualActivity: String?

将来的に追加する可能性：

- intendedTask: String
- category: String
- context: String
- device: String
- locationType: String
- focusLevel: Int

## 画面構成

### 1. ホーム画面 / 今日のアンカー

毎日最初に見る画面。

表示するもの：

- 今日のアンカー
- 今の重点テーマ
- 次の一手
- 今日の積み上げ時間
- 今日の迷い回数
- 棚上げ中の項目
- 作業開始ボタン
- 迷ったボタン
- チェックインボタン

役割：

迷ったときに戻る場所。
今やることを思い出す画面。

### 2. 迷いログ画面

迷った内容を保存・確認する画面。

保存するもの：

- 迷った内容
- 作成日時
- 関連テーマ
- 迷いの強さ
- 結果

最初は content と createdAt だけでよい。

最初の一覧画面は閲覧専用でよい。
削除、編集、検索、分類、強さの記録は後回しにし、まずは「預けた迷いを見返せる」ことを優先する。

### 3. 棚上げリスト画面

今はやらないが、捨てない選択肢を保存する画面。

保存するもの：

- 棚上げしたこと
- 理由
- 作成日時
- 見直し日

### 4. 作業タイマー画面

何にどれだけ時間を投下したかを記録する画面。

保存するもの：

- 作業名
- 開始時刻
- 終了時刻
- 作業時間

最初はホーム画面内の簡易タイマーでもよい。

### 5. チェックイン画面

1日の終わりに軽く記録する画面。

保存するもの：

- 達成度 1〜5
- 納得度 1〜5
- 迷いの強さ 1〜5
- 一言メモ
- 作成日時

## データモデル案

### DoubtLog

迷いログ。

Fields:

- id: UUID
- content: String
- createdAt: Date

将来的に追加する可能性：

- intensity: Int
- relatedTheme: String
- result: String

### ShelfItem

棚上げ項目。

Fields:

- id: UUID
- title: String
- reason: String
- createdAt: Date
- reviewDate: Date?

### CheckIn

日次チェックイン。

Fields:

- id: UUID
- achievementScore: Int
- satisfactionScore: Int
- doubtScore: Int
- memo: String
- createdAt: Date

### ActivitySession

作業・行動時間ログ。

Fields:

- id: UUID
- title: String
- startedAt: Date
- endedAt: Date?
- durationSeconds: Int
- result: String?
- actualActivity: String?

将来的に追加する可能性：

- intendedTask: String
- category: String
- context: String
- device: String
- locationType: String
- focusLevel: Int

## デザイン方針

- 落ち着いた雰囲気
- 自分を責めない言葉遣い
- 余白を多めにする
- カードUI中心
- 情報を詰め込みすぎない
- 1画面で「戻ってこられる感覚」を出す
- 派手な色や過度なアニメーションは不要

## 文言方針

避ける表現：

- 達成率が低いです
- もっと頑張りましょう
- 未完了タスクがあります
- 失敗しました

使いたい表現：

- 迷いはここに預ける
- 次の一手に戻る
- 今日は迷いが多い日でした
- 記録すれば、次に使えます
- 今は考えるより進める時間です
- 棚上げして、今やることに戻る

## 実装順

1. 静的ホーム画面
2. DoubtLog モデル作成
3. 迷ったボタンから sheet で迷い入力
4. 今日の迷い回数をホームに反映
5. 迷いログ一覧画面
6. ShelfItem モデル作成
7. 棚上げリスト追加・一覧
8. ActivitySession モデル作成
9. 作業開始・停止・積み上げ時間表示
10. セッション終了後フィードバック画面
11. 代替行動ログ保存
12. CheckIn モデル作成
13. チェックイン入力画面
14. 簡単な履歴表示

## Codexへの基本方針

- 変更範囲は小さくする
- 一度に多くの機能を実装しない
- 既存のUIを大きく崩さない
- まずビルドが通ることを優先する
- 外部ライブラリは使わない
- SwiftDataで保存する
- 複雑な抽象化はまだしない
- まず自分が使える最小実装を優先する

## 現在の優先順位

最優先は、以下の体験を作ること。

迷ったときにアプリを開く  
→ 今日のアンカーを見る  
→ 迷いをログに預ける  
→ 今やることに戻る  

この体験が成立するまでは、他の機能を増やしすぎない。
